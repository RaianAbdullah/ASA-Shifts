# Build a Secure Workforce Attendance Mobile Application From Scratch

Act as a senior full-stack software engineer, mobile developer, cybersecurity engineer, database architect, UI/UX designer, and software tester.

Build a complete application from scratch called:

# ASA Workforce

The full academic project name is:

**Secure Workforce Mobile Application System**

This application will be used by a government workplace to manage:

* Employees
* Departments
* Weekly work schedules
* Weekend duty schedules
* Attendance check-in
* Vacation requests
* Managers
* Administrators
* Security logs
* Notifications

The current workplace uses WhatsApp to communicate changing weekly schedules. The new application must replace this process with a secure and organised mobile system.

Security must be designed from the beginning. Do not build the application first and add security later.

# 3. User Registration and Approval Process

Implement this exact account workflow.

## Step 1: Registration

The employee enters:

* First name
* Last name
* Employee number
* Work email
* Phone number
* Password
* Password confirmation
* Requested department, where appropriate

After registration, set the account status to:

`PENDING_OTP`

## Step 2: OTP Verification

Send a one-time verification code through email or SMS.

After successful OTP verification, change the status to:

`PENDING_ADMIN_APPROVAL`

## Step 3: Admin Review

An authorised Admin reviews the account.

The Admin can:

* Approve the user
* Reject the user
* Request corrected information

After approval, change the status to:

`ACTIVE`

Only an `ACTIVE` account can log in.

Support these account statuses:

* `PENDING_OTP`
* `PENDING_ADMIN_APPROVAL`
* `ACTIVE`
* `REJECTED`
* `SUSPENDED`
* `LOCKED`
* `DEACTIVATED`
* `PASSWORD_RESET_REQUIRED`

Every account status change must be recorded in the audit log.

A user must never be able to:

* Approve their own account
* Assign their own role
* Change their own department permissions
* Activate a suspended account

# 4. Authentication Requirements

Implement:

* Secure registration
* OTP verification
* Admin approval
* Login
* Logout
* Access tokens
* Refresh tokens
* Refresh-token rotation
* Session revocation
* Password reset
* Change password
* Account lockout
* Active session management
* New-device notification
* Force logout from all devices
* Optional biometric unlock on the mobile device later

Use a short-lived JWT access token.

Recommended initial configuration:

* Access token: approximately 10–15 minutes
* Refresh token: approximately 3 days
* OTP: approximately 5 minutes

Keep these configurable through environment variables.

Do not store raw refresh tokens in the database. Store a secure hash of each refresh token.

When a refresh token is used:

1. Validate it.
2. Revoke the old refresh token.
3. Create a new refresh token.
4. Detect reuse of a revoked token.
5. Revoke the entire token family when reuse is detected.
6. Create a security audit event.
7. Require the user to log in again.

Revoke active sessions when:

* The user logs out
* The password changes
* A password reset succeeds
* The account is suspended
* The account is deactivated
* Refresh-token reuse is detected
* An administrator forces logout
* A serious security event occurs

# 7. Attendance Check-In

The initial application requires check-in only.

Do not add check-out unless requested later.

An attendance record should contain:

* Attendance ID
* Employee
* Department
* Attendance date
* Check-in timestamp
* Related schedule
* Attendance classification
* Verification method
* QR verification result, if used
* Location verification result, if used
* Device session reference
* Creation timestamp
* Administrative correction details, if applicable

The backend must generate the official check-in time.

Never trust:

* The mobile device time
* A time submitted by the frontend
* The frontend’s attendance classification
* A user ID submitted without checking ownership

Prevent:

* Duplicate check-in on the same attendance day
* Checking in for another employee
* Backdated check-in by employees
* Check-in from suspended users
* Check-in outside authorised conditions
* Reuse of an old attendance request
* Modification of another user’s attendance record

Use database constraints and backend validation to prevent duplicate attendance.

## Attendance Classifications

Support:

* `ON_TIME`
* `LATE`
* `UNSCHEDULED`
* `WEEKEND_DUTY`
* `APPROVED_ABSENCE`
* `VACATION`
* `MISSING`
* `MANUALLY_CORRECTED`

The backend must calculate the classification using the approved schedule and server time.

Do not allow the mobile app to decide whether an employee is late or on time.

# 8. Attendance Verification Options

Build the core attendance system first.

Then prepare optional support for:

* Dynamic QR check-in
* Geofencing
* Device risk signals

These are additional security signals, not perfect proof that the employee is physically present.

## Dynamic QR Check-In

If implemented:

* Generate a short-lived QR challenge
* Sign the challenge securely
* Set a short expiration time
* Bind it to a workplace, date, or attendance session
* Verify it on the backend
* Prevent replay
* Mark it as used where appropriate
* Do not include sensitive employee information in the QR code
* Record failed and successful attempts

Do not use a permanent QR code printed on a wall.

## Geofencing

If implemented:

* Request location only during check-in
* Explain why location is needed
* Validate the location on the backend
* Define an allowed radius
* Consider GPS accuracy
* Detect obvious mock-location risks where practical
* Do not continuously track employees
* Do not store precise location longer than necessary
* Store the verification result rather than unnecessary location history where possible
* Provide an authorised exception process when GPS is unavailable

Do not claim that geofencing completely prevents attendance fraud.

1. Employee submits a vacation request.
2. Direct or main manager reviews it.
3. Manager approves or rejects it.
4. If approved, notify:

   * The Department Manager
   * The responsible administrative person
5. Update the employee’s schedule or attendance classification where required.

A vacation request should include:

* Request ID
* Employee
* Start date
* End date
* Vacation type
* Reason
* Optional attachment
* Status
* Submitted timestamp
* Reviewed timestamp
* Reviewer
* Decision notes

Statuses should include:

* `PENDING`
* `APPROVED`
* `REJECTED`
* `CANCELLED`

Validate that:

* End date is not before start date
* An employee cannot approve their own request
* Unauthorised managers cannot review the request
* Overlapping requests are handled correctly
* Attachments are protected

# 16. Essential Security Requirements

Security is the most important part of this system.

Use secure-by-design, privacy-by-design, least privilege, deny-by-default, and defence-in-depth principles.

Follow relevant guidance from:

* OWASP Application Security Verification Standard
* OWASP API Security
* OWASP Mobile Application Security
* NIST authentication guidance
* Current Spring Security recommendations

Do not merely mention security standards. Implement and test the controls.

# 17. Password Security

Implement:

* BCrypt or Argon2 password hashing
* Strong adaptive hashing configuration
* Minimum password requirements
* Support for long passphrases
* Maximum reasonable input length
* Compromised-password protection where practical
* Secure password change
* Secure password reset
* Password history if required by policy
* Session revocation after password changes

Never:

* Store plaintext passwords
* Encrypt passwords reversibly
* Log passwords
* Send passwords by email
* Allow administrators to view passwords
* Hardcode passwords
* use a weak default password

Password-reset tokens must:

* Be cryptographically random
* Be single-use
* Expire quickly
* Be stored as hashes
* Be invalidated after use
* Be invalidated when a newer token is generated

---

# 18. OTP Security

OTP codes must:

* Be generated using a cryptographically secure random generator
* Expire after a short period
* Be single-use
* Have a maximum number of attempts
* Have a resend cooldown
* Be rate-limited
* Invalidate the previous OTP when a new OTP is created
* Be stored as a secure hash
* Never appear in production logs
* Never be included in production API responses

Use generic responses to reduce account enumeration.

For local development, create a clearly separated development-only OTP delivery method.

The development OTP shortcut must be disabled automatically in production.

Implement role-based and ownership-based authorisation on the backend.

Requirements:

* Deny access by default
* Validate the authenticated user for every protected request
* Do not trust roles sent by the frontend
* Employees can access only their own records
* Managers can access only authorised employees
* Department Managers can access only their department
* Admin endpoints reject employees and ordinary managers
* Only specific roles can modify attendance
* Only authorised roles can publish schedules
* Only authorised managers can approve vacation requests
* Users cannot assign their own role
* Users cannot approve themselves
* Users cannot change protected account fields

Protect against:

* Horizontal privilege escalation
* Vertical privilege escalation
* Insecure direct object references
* Broken function-level authorisation
* Mass assignment

Create automated tests for every major permission boundary.

* Login
* Registration
* OTP generation

* Attendance check-in

# 22. JWT and Refresh-Token Security

JWT requirements:

* Strong signing keys
* Secrets loaded from secure environment configuration
* Signature validation
* Expiration validation
* Issuer validation
* Audience validation
* Token-type validation
* Minimal claims
* No sensitive personal information in tokens
* Short expiration
* Server-side status verification

Do not rely only on the role stored in an old token.

For important requests, verify the current user status and permissions from the database or a secure current-authorisation source.

An account that becomes suspended must lose access even if an access token has not yet naturally expired.

Refresh tokens must:

* Be cryptographically random
* Be stored only on the client as a raw value
* Be stored as hashes on the server
* Be rotated after every use
* Belong to a token family
* Support reuse detection
* Support session revocation
* Support device-session visibility

Do not put refresh tokens inside ordinary application logs.

# 36. How You Must Work

Do not attempt to generate the entire application in one uncontrolled change.

Work stage by stage.

At the beginning of every stage:

1. State the goal.
2. List the files you will create or modify.
3. Explain the security considerations.
4. Explain how the stage will be tested.

After every stage:

1. Run the code.
2. Run relevant tests.
3. Fix compilation errors.
4. Fix failing tests.
5. Explain what was completed.
6. Explain how to run it.
7. Explain any new command, window, simulator, process, warning, or configuration.
8. Update the documentation.
9. Create a clear Git commit.

Do not move to the next stage while the current stage has serious errors.

Use small, meaningful Git commits such as:

```text
chore: initialise workforce project structure
feat: add secure employee registration
feat: implement OTP verification
feat: add admin account approval
feat: implement rotating refresh tokens
test: add authentication integration tests
security: add refresh-token reuse detection
```
